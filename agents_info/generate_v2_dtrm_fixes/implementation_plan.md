## Problem Statement
When the user opens `ter_elysium.hod`, makes edits (rename joints, delete nodes, add turret barrel), and saves as `ter_elysium_edited.hod`, the resulting file is structurally corrupted. Re-opening it in the app shows incorrect/broken nodes.

Binary analysis confirmed that the **user's edits were correctly serialized** (renamed joints, deleted nodes, added barrel all visible in the hex dump), but the **surrounding structural metadata is wrong**, causing the parser to misinterpret the file on reload.

## Root Cause Analysis â€” 5 Bugs Found

### Bug 1: HIER `first_val` is hardcoded to `0xFFFFFF00`
**File**: [hod.rs](file:///run/media/system/Data/SteamLibrary/steamapps/common/Homeworld%20347380/GBXTools/WorkshopTool/mod-tools/HODEditorJS/parser/src/hod.rs#L2902)

The `generate_v2_from_model` function hardcodes the HIER header to `0xFFFFFF00`, while the correct encoding is `0xFFFFFF00 | ((-joint_count) & 0xFF)`.

| File | first_val | Joint Count |
|------|-----------|-------------|
| Original | `0xFFFFFFE2` | 30 (-30 & 0xFF = 0xE2) |
| Edited | `0xFFFFFF00` | 29 (should be 0xFFFFFFE3) |

The `save_edits` function at line 3278 correctly computes this, but `generate_v2_from_model` does not.

**Fix**: Replace line 2902's hardcoded value with the same computation from `save_edits`.

---

### Bug 2: HIER chunk written as `ChunkType::Normal` instead of `ChunkType::Form`
**File**: [hod.rs](file:///run/media/system/Data/SteamLibrary/steamapps/common/Homeworld%20347380/GBXTools/WorkshopTool/mod-tools/HODEditorJS/parser/src/hod.rs#L2927-L2933)

In `generate_v2_from_model`, the HIER chunk is created with `ChunkType::Normal` (NRML), but the original file uses `FORM HIER`. The parser at line 2207 checks `(first_val & 0xFFFFFF00) == 0xFFFFFF00` to detect HOD 2.0, which works regardless, but the engine itself expects `FORM HIER`.

**Evidence**: Original = `FORM HIER size=1790`, Edited = `NRML HIER v=0 size=1736`

**Fix**: Change `chunk_type` to `ChunkType::Form` at line 2929.

---

### Bug 3: Rotation data lost â€” zeros written instead of actual rotation values
**File**: [hod.rs](file:///run/media/system/Data/SteamLibrary/steamapps/common/Homeworld%20347380/GBXTools/WorkshopTool/mod-tools/HODEditorJS/parser/src/hod.rs#L2918-L2925)

The `generate_v2_from_model` function writes hardcoded `0.0, 0.0, 0.0` for rotation and `1.0, 1.0, 1.0` for scale, completely ignoring the actual parsed joint data.

**Evidence**: Original EngineNozzle3 has `rot=(0.0, 3.142, 0.0)`, but edited version outputs `rot=(0.0, 0.0, 0.0)`. The `save_edits` function at line 3285-3302 correctly reads from `joint.rotation` / `joint.scale`.

**Fix**: Use the same logic as `save_edits` â€” read from `joint.position`, `joint.rotation`, `joint.scale` (with `decompose_matrix` fallback).

---

### Bug 4: BURN chunks consolidated into single NRML BURN instead of individual default BURN chunks
**File**: [hod.rs](file:///run/media/system/Data/SteamLibrary/steamapps/common/Homeworld%20347380/GBXTools/WorkshopTool/mod-tools/HODEditorJS/parser/src/hod.rs#L2966-L2988)

The original file has 8 separate `BURN` chunks (each with `ChunkType::Default` and 100 bytes). The `generate_v2_from_model` function consolidates ALL burns into a single `NRML BURN v=3` chunk.

**Evidence**: Original = 8Ă— `BURN size=100`, Edited = 1Ă— `NRML BURN v=3 size=608`

**Fix**: Write individual `BURN` chunks with `ChunkType::Default`, matching the `save_edits` approach.

### Bug 5: NAVL data not regenerated by `generate_v2_from_model`
**File**: [hod.rs](file:///run/media/system/Data/SteamLibrary/steamapps/common/Homeworld%20347380/GBXTools/WorkshopTool/mod-tools/HODEditorJS/parser/src/hod.rs#L2991-L2999)

The `generate_v2_from_model` preserves DTRM children that are NOT `HIER`, `MRKR`, or `BURN` â€” so NAVL, MRKS, KDOP, COLD, SCAR are preserved from the original. This means if navlights were renamed/edited in the UI, those changes are **lost** since the original NAVL data is kept verbatim.

The user renamed navlights "EngineNozzle1" â†’ "e1" and "EngineNozzle2" â†’ "e2", but the NAVL chunk still references the original names from the preserved original data. The `save_edits` function at line 3354-3371 correctly regenerates NAVL from the model.

**Fix**: Regenerate NAVL data from `updated_model.nav_lights` instead of preserving original, matching the `save_edits` approach.

## Proposed Fix

The simplest and most correct fix: **make `generate_v2_from_model` delegate DTRM generation to the same logic used by `save_edits`**. Specifically:
1. Fix the `first_val` computation
2. Fix the HIER chunk type to `ChunkType::Form`
3. Use `joint.position`/`.rotation`/`.scale` for transform data
4. Write individual BURN chunks
5. Regenerate NAVL from model data instead of preserving original
