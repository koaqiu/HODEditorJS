# DAEnerys OBJ to DAE Pipeline Analysis

To ensure no details are skipped regarding how HODOR reads meshes, this document outlines the exact pipeline DAEnerys uses to convert OBJ files to DAE before HODOR consumes them. This analysis was performed by inspecting the C# source code of the `DAEnerys` mod tool.

## 1. OBJ Loading (Assimp)

DAEnerys uses the `Assimp` library (`AssimpContext.ImportFile`) to load OBJ files in `ObjImporter.cs`. 

Crucially, it imports the mesh with the following flags:
```csharp
Scene obj = importer.ImportFile(path, PostProcessSteps.JoinIdenticalVertices | PostProcessSteps.Triangulate | PostProcessSteps.ValidateDataStructure);
```

### Vertex Splitting & Flattening
Because the OBJ format uses separate index streams for positions (`v`), texture coordinates (`vt`), and normals (`vn`), a single position might be used by multiple faces with different UV coordinates. 

When Assimp loads the file with these flags, it automatically resolves these multi-indexed structures by **splitting** vertices that have the same position but different UVs or Normals. The output is a "flat" mesh where the `Vertices`, `Normals`, and `TextureCoordinateChannels` arrays are all the exact same length, and a single shared index buffer can address all three attributes simultaneously.

## 2. Mesh Consolidation by Material

After Assimp parses the file, DAEnerys iterates through the resulting sub-meshes and groups them by their assigned `Material` (`obj.Materials[mesh.MaterialIndex]`). 

It concatenates the geometry data of all meshes that share the same material into a single `Mesh` object. During this process, it appends the vertices, normals, and UVs, and shifts the incoming face indices by the existing `VertexCount` to maintain structural integrity.

## 3. DAE Export (COLLADA)

When exporting to DAE (`Exporter.cs`), DAEnerys uses LINQ to XML to write the COLLADA 1.4.1 format.

### Shared Indexing in `<triangles>`
Because Assimp has already flattened the vertices, DAEnerys's DAE exporter writes `<source>` arrays for Position, Normal, and UVs sequentially, and then defines the `<triangles>` block with a shared index format. 

For each semantic input (VERTEX, NORMAL, TEXCOORD), it writes the **exact same index** from the mesh's index buffer:
```csharp
for (int f = 0; f < mesh.IndexCount; f += 3)
{
    for (int i = 0; i < 3; i++)
    {
        for (int c = 0; c < inputCount; c++)
        {
            faces.Append(" " + (mesh.Indices[f + i] + indexOffset));
        }
    }
    faces.AppendLine();
}
```
If a mesh has Position, Normal, and UV, the resulting `<p>` array will look like:
`<p>0 0 0 1 1 1 2 2 2 ...</p>`

## Conclusion: Implications for HODOR

Because DAEnerys outputs DAE files where every vertex attribute uses the exact same index, **HODOR expects and relies on this pre-flattened vertex structure**. 

When we feed DAE files directly to HODOR (or replicate its DAE parsing), HODOR does not need to perform complex multi-index vertex splitting itself—it just reads the interleaved attributes directly as they appear sequentially in the `<source>` arrays or assumes the `<p>` array indices are perfectly mirrored across all attributes. 

This confirms that the vertex splitting logic happens **upstream** in DAEnerys via Assimp, not inside HODOR. Our replication efforts in `parser/src/compiler.rs` correctly mirror this by reading the DAE file and relying on the pre-split vertices generated by the DAEnerys pipeline.
