# HOD 2.0 Editor Architecture Overview

This document provides a high-level view of the HODEditorJS project objectives, workflow, and data pipeline. If you are a new contributor or agent, understanding this unified vision is essential before diving into individual components or hex codes.

## 1. Project Objective

The core objective of this project is to create **HOD 2.0 files** for Homeworld Remastered via a visual UI editor (`HODEditorJS`). 

Currently, the official modding pipeline relies on a complex chain of command-line tools (OBJ -> DAEnerys -> HODOR). Our goal is to streamline this by providing a unified editor that handles the entire pipeline internally, while generating files that are 100% compatible with the Homeworld Remastered game engine.

## 2. Editor Workflow

In the editor, users will have multiple ways to create and manipulate HOD data:

1. **Node Authoring**: Users can manually create the HOD structure by adding specific Nodes (e.g., joints, navlights, engine burns, collision meshes) using the editor's UI.
2. **Asset Import**: Users can import raw 3D assets:
   - **Meshes** from `.OBJ` files.
   - **Textures** from `.TGA` files for material creation and texture mapping.
3. **HOD Editing**: Users can open existing `HOD 1.0` or `HOD 2.0` files, fully edit their contents (add/remove nodes, modify materials), and then save the result back out as a valid `HOD 2.0` file.
4. **DAE Import**: Users can open a `.DAE` (COLLADA) file, edit its contents in the UI, and save it as a `HOD 2.0` file.

## 3. The Data Pipeline (Why We Reverse-Engineer)

When the editor saves to HOD 2.0, the resulting file must be **fully compatible** with the game engine. 

To guarantee this compatibility without relying on closed-source executables, we are completely replicating the `OBJ -> DAE -> HODOR` pipeline in Rust (`parser/src/`).

### The Original Pipeline:
1. **Source Assets (`.OBJ` / `.TGA`)** are authored by artists.
2. **DAEnerys** imports the OBJ (resolving multi-index structures into flattened arrays), groups them by material, and exports a pre-flattened `.DAE` file.
3. **HODOR** reads the `.DAE` file and the `.TGA` textures. It then:
   - Computes tangent spaces and performs selective vertex deduplication.
   - Uses **Microsoft Xpress (LZ77)** to compress the textures, meshes, and faces.
   - Serializes everything into the final `HOD 2.0` binary format.

### Our Replicated Pipeline (`HODEditorJS`):
Our editor aims to **skip the intermediate files** while perfectly mimicking the data transmutation steps. When a user clicks "Save to HOD 2.0", the editor must:
1. Flatten and process the imported OBJ/DAE meshes exactly as DAEnerys does (relying on pre-flattened DAE logic).
2. Calculate tangent spaces and deduplicate vertices exactly as HODOR does.
3. Compress the resulting payload using our replicated, byte-accurate **Xpress Compressor**.
4. Serialize the final `HOD 2.0` layout exactly as the game engine expects it.

By meticulously documenting and verifying each of these steps (as tracked in `PROGRESS.md`), we ensure the HODs generated by our editor will render flawlessly in-game.
